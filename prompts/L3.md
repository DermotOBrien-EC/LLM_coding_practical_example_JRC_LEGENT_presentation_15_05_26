# TASK

You are running a small, reproducible **forecasting bake-off** for German
national hourly electricity load. Treat this exercise like a short methods
section of a peer-reviewed paper: multiple model classes are fit on the same
data, evaluated on the same held-out test window, and compared on the same
metrics. Produce publication-grade figures, a transparent metrics table, and
a written discussion of which model wins and where each one fails.

The deliverable is a self-contained study an external reviewer could read in
under five minutes and reproduce in under thirty.

# BACKGROUND

## Data

- File: `opsd_de_load.csv` (in the current working directory). Columns:
    - `utc_timestamp` — ISO-8601, hourly, UTC
    - `DE_load_actual_entsoe_transparency` — load in megawatts
- Coverage: 2015-01-01 00:00 UTC to 2020-09-30 23:00 UTC, hourly,
  exactly 50,400 rows, no NaN, no gaps.
- Source: Open Power System Data, derived from ENTSO-E Transparency Platform.

## Models to compare

Compare **all six** of the following, end-to-end. The point of this exercise
is the bake-off, not picking a favourite a priori.

1. **Seasonal-naive baseline** — the forecast for hour *t* is the observed
   load at hour *t* − 168 h (same hour, one week earlier). Anchor for
   relative comparisons; everything else should beat this.

2. **SARIMA** — classical statistical approach. Fit via
   `pmdarima.auto_arima` with weekly seasonality (m=168 is impractical;
   use m=24 daily seasonality and let weekly structure emerge from
   differencing) **or** via `statsmodels` SARIMAX with explicit
   `(p,d,q)(P,D,Q,s)` orders chosen by the validation MAPE.

3. **Prophet with German holidays** — via `darts.models.Prophet`. Use
   `country_holidays="DE"` (or pass holidays explicitly via the
   `holidays.Germany()` package). Daily + weekly + yearly seasonality.

4. **LightGBM on engineered features** — via `lightgbm.LGBMRegressor`
   directly (cleaner than darts wrapper for feature work). Features:
     - Time index: `hour`, `day_of_week`, `month`, `is_weekend`,
       `is_public_holiday_de` (use `holidays.Germany()` for the indicator)
     - Lag features from the load itself:
       `lag_24h`, `lag_168h` (1 week), `lag_8760h` (~1 year, day-of-week
       aligned via the 364-day offset if cleaner)
     - Rolling-window features: 24 h and 168 h moving averages and stds
       of the load, computed *only* on data ≤ each row's timestamp.
   For each test-window hour, all required lags must be present in the
   data (they are: 8760 h before 2020-01-01 is 2018-12-31 16:00 UTC,
   well inside the training period).

5. **N-BEATS** — univariate deep MLP-based forecaster via
   `darts.models.NBEATSModel`. Use the default
   stack configuration; `input_chunk_length=168` (one week of context),
   `output_chunk_length=168` (the forecast horizon), at most ~30 epochs
   with early stopping on validation MAPE.

6. **PatchTST** — univariate transformer forecaster via
   `darts.models.TSMixerModel` or `darts.models.PatchTSTModel` if the
   installed darts version exposes it; otherwise substitute another
   transformer-based univariate model from the darts catalogue and record
   the substitution in `transcript.md`. Same input/output chunk lengths
   as N-BEATS. At most ~30 epochs with early stopping.

## Allowed inputs

- The load series itself (`DE_load_actual_entsoe_transparency`).
- Features derived from the `utc_timestamp` column without consulting any
  external dataset: `hour`, `day_of_week`, `month`, `is_weekend`,
  `is_public_holiday_de` (German federal holidays via the `holidays`
  package — knowable a priori, not exogenous data).
- Lag features and rolling-window features computed from the load itself.

## Forbidden inputs

- **External data**: temperature, weather, gas prices, day-ahead market
  prices, EUA carbon prices, anything not in `opsd_de_load.csv`.
- If you find yourself wanting to bring in such data, stop and record why
  in `transcript.md`. The univariate constraint is deliberate: this study
  isolates the temporal-structure component of forecast skill, not the
  weather-dependence component.

## Splits

- **Train (initial fit and hyperparameter sweeps):**
  2015-01-01 00:00 to 2019-09-30 23:00 (≈4.75 years, ~41,500 rows).
- **Validation (hyperparameter / model-order selection):**
  2019-10-01 00:00 to 2019-12-31 23:00 (~2,200 rows).
- **Test (held out, never touched during fitting or selection):**
  2020-01-01 00:00 to 2020-01-07 23:00 (168 hours).

Workflow per model:
1. Fit on Train.
2. Select hyperparameters by evaluating on Validation.
3. Refit the chosen configuration on **Train + Validation** combined
   (2015-01-01 to 2019-12-31). This is standard practice — final model
   uses all data prior to the test window.
4. Forecast the 168 test hours.
5. Score against the held-out test observations.

The naive baseline does not need step 2 (no hyperparameters); it is fit
implicitly on whatever lags it needs.

## Metrics

Compute and report for **every model** on the test window:

- **MAPE** — mean absolute percentage error (primary metric).
- **RMSE** — root mean square error in MW.
- **MAE** — mean absolute error in MW.
- **Holiday-vs-working-day MAPE breakdown**: MAPE restricted to Jan 1
  (Wednesday, German public holiday) and MAPE on Jan 2–7 separately.
  This is where models that don't model holidays will be exposed.

For the **winning model** (lowest test MAPE):

- **80% and 95% prediction-interval coverage** — fraction of actuals
  inside each interval. Report both nominal and actual coverage.
- **Pinball loss** at quantiles 0.1, 0.5, 0.9.

Probabilistic forecasts are required for SARIMA, Prophet, N-BEATS, and
PatchTST (their libraries support sampling or analytic intervals).
LightGBM can use `LGBMRegressor` with quantile objective at the 0.1, 0.5,
0.9 quantiles (three fits) to produce intervals. The naive baseline
does not produce intervals.

## Figures

All figures: **300 dpi, sans-serif (matplotlib default), clear axis labels
with units in parentheses, non-overlapping legends, consistent `tab10` or
`Set2` qualitative palette across figures so each model has the same colour
in every plot.** Time-series figures: 11 × 6 in. Square comparisons: 6 × 6 in.

1. **`01_overview.png`** — the full load series 2015-01-01 → 2020-09-30 in
   light grey, with the 168-hour test window highlighted in red. One panel,
   gives the reader spatial context.

2. **`02_forecast_comparison.png`** — small multiples, 2 × 3 grid (one
   subplot per model). Each subplot shows the test-week observed values
   in black and that model's point forecast in its assigned colour. Same
   y-axis range across all six panels. Title per subplot: model name +
   its MAPE. Saves eye-strain vs overlaying six lines.

3. **`03_metric_comparison.png`** — bar chart of MAPE, RMSE, MAE per model
   (three grouped bars per model, six models). Sort by MAPE ascending. Show
   the numeric value above each bar.

4. **`04_per_day_mape.png`** — heatmap or grouped bars of per-day MAPE
   (7 days × 6 models). This is where the holiday weakness becomes visible:
   most models should have a tall bar on Jan 1.

5. **`05_winner_with_intervals.png`** — winning model's forecast on the
   test week with 80% (lighter band) and 95% (lighter still) prediction
   intervals; observed in black; point forecast in the winner's colour.
   Caption: model name + test MAPE + actual 80% / 95% coverage rates.

6. **`06_residuals.png`** — winning model's residuals on the test window,
   broken down by hour-of-day (left subplot, box plot or violin) and
   day-of-week-of-week-in-test (right subplot). Tells the reader where the
   winner systematically over- or under-predicts.

7. **`07_feature_importance.png`** *(produce if LightGBM is in the top 2,
   not just if it wins)* — `LGBMRegressor` feature importances (gain),
   sorted descending.

8. **`08_decomposition.png`** *(produce if Prophet is in the top 2)* —
   trend, weekly, yearly components from the fitted Prophet model.
