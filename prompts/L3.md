# TASK

Build a Python script that produces a 168-hour (7-day) ahead point and
probabilistic forecast for German national electricity load using only past
values of the load series (univariate). Train, validate on a hold-out
window, evaluate on a separate held-out test window, then save the
artifacts specified below.

# BACKGROUND

## Data

- File: `../../data/opsd_de_load.csv`. Columns:
    - `utc_timestamp` (ISO-8601, hourly, UTC)
    - `DE_load_actual_entsoe_transparency` (megawatts)
- Coverage: 2015-01-01 00:00 UTC to 2020-09-30 23:00 UTC, hourly resolution,
  ~50,000 rows.
- Source: Open Power System Data, derived from ENTSO-E Transparency Platform.

## Modelling choice

Pick exactly one of the following model classes and justify your choice in
2–3 sentences at the top of `transcript.md` before you write any code:

- **SARIMA** (via `pmdarima.auto_arima` or `statsmodels` SARIMAX) —
  handles daily and weekly seasonality through seasonal differencing.
- **Prophet** (via `darts.models.Prophet`) — daily / weekly / yearly
  seasonality decomposition with prediction intervals out of the box.
- **N-BEATS** or **N-HiTS** (via `darts.models.NBEATSModel` /
  `darts.models.NHiTSModel`) — univariate deep models that learn
  multi-scale temporal structure end-to-end.

Implement only one model. The point is not to benchmark — it is to produce
one well-validated forecast end-to-end.

## Splits

- **Train (initial fit and hyperparameter sweeps):**
  2015-01-01 00:00 to 2019-09-30 23:00 (≈4.75 years, ~41,500 rows).
- **Validation (hyperparameter / model-order selection):**
  2019-10-01 00:00 to 2019-12-31 23:00 (~2,200 rows).
- **Test (held out, never touched during fitting or selection):**
  2020-01-01 00:00 to 2020-01-07 23:00 (168 hours, ~168 rows).

Workflow:
1. Fit candidate models on **Train** only.
2. Select hyperparameters / model order by evaluating on **Validation**.
3. **After selection is finalised, refit the chosen model on Train + Validation
   combined (2015-01-01 to 2019-12-31).** This is the standard practice — you
   want the final model fitted on the most recent observations available.
4. Forecast the next 168 hours (2020-01-01 to 2020-01-07) and compute MAPE
   against the held-out test window.

The 168-hour test window IS the forecast you produce and evaluate. There is
no separate "final forecast" beyond the data.

## Metric

Mean Absolute Percentage Error (MAPE) on the test window. Report it
numerically in `metrics.json` and in the figure caption.

## Figure specification

Save to `figures/forecast.png`, 300 dpi, A4-landscape proportions
(roughly 11 × 6 inches).

- x-axis: time, formatted `YYYY-MM-DD HH`.
- y-axis: load in MW.
- Plot the last 30 days of training data in light grey.
- Plot the 168-hour test window observed values as a thin black line.
- Plot the model forecast over the test window as a blue line.
- Shade the 95% prediction interval as a wider, lighter blue band.
- Shade the 80% prediction interval as a narrower, darker blue band.
- Vertical dotted line at the train → test boundary, labelled
  "test window starts".
- Figure caption (in the figure, below the plot):
  `MAPE on 2020-01-01 to 2020-01-07 168-hour test = X.XX %`.

## Outputs to produce in your working directory

- `code/forecast.py` — the script.
- `figures/forecast.png` — the figure as specified.
- `metrics.json` — single JSON object with the schema in `AGENTS.md`.
- `transcript.md` — Markdown log of your decisions (model choice and
  justification at the top, hyperparameters, any issues hit, time spent).

# DO NOT

- Do not use the test window (2020-01-01 to 2020-01-07) for any model fitting,
  hyperparameter selection, or model-class choice. The test window is reserved
  for the final evaluation only.
- During initial fitting and hyperparameter sweeps, do not use data after
  2019-09-30 23:00. The validation window (2019-10-01 to 2019-12-31) is only
  for scoring candidate hyperparameter / model-order choices. After selection
  is finalised, refitting on train + validation combined is required, not
  optional.
- Do not introduce exogenous variables (temperature, calendar holidays,
  weather, day-of-week dummies derived from external sources). This is a
  deliberate univariate constraint.
- Do not silently drop or impute missing hours. If gaps exist, report the
  count in `transcript.md` and pick an explicit, justified strategy.
- Do not skip the validation step. Hyperparameter or model-class choice that
  touches the test window invalidates the demonstration.
- Do not edit any file under `prompts/` or any file under `runs/` outside
  your own level's directory.
- Do not produce a figure that lacks the train/test boundary line or the
  MAPE caption.
- Do not install additional Python packages. If something is missing, record
  the gap in `transcript.md` and work around it or stop.
